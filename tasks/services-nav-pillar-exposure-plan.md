# ヘッダー・フッター・/services ピラー露出設計 — 2026-07-23

## 結論（1行）

新設6ピラー（group-home / kaigo / minpaku / inshokuten / office / souzoku/taiwan）はsitemap優先度こそ高い（0.8〜0.9）が、ヘッダー・フッター・/servicesという一次導線からはほぼ到達不可能。4カテゴリ（相続／グループホーム開設／投資事業用・開業支援／外国人）に整理し、ヘッダーのメガメニュー化・フッターの4カラム化・/servicesのカード拡張の3点セットで露出させる。

## 現状調査サマリー

- **ヘッダー**（`src/components/layout/TenantLayout.tsx` 26〜33, 181〜358行目）：サービス／会社概要／コラム／アクセス／お問い合わせの5項目フラット構成。ドロップダウンなし。
- **フッター**（同ファイル 56〜163, 360〜634行目）：3カラム（サービス／会社情報／その他）。「サービス」列の4項目は**すべて`/services`への同一リンク**（アンカーid未実装のための暫定対応）。
- **/servicesページ**（`src/app/(realestate)/services/page.tsx`）：リポジトリ上（2026-07-10 浦松承認・コミット`4db9cce`、mainマージ済み）は相続／投資事業用／外国人の3カードのみ。新設6ピラー・/souzoku/akiya・/faq・/jirei等は未掲載。
- **`src/lib/cross-links.ts`**：不動産⇄行政書士⇄社労士の分離受任リンク表（C1〜C14）。新設6ピラーは対象外。ページ間リンクは各ページの`internalLinks`propと`faqJa.ts`で個別実装のみ。
- **孤立ページ（ヘッダー/フッター/servicesのいずれからも到達不可）**：`/kaigo`・`/minpaku`はほぼ完全孤立（FAQ経由とllms.txtのみ）。`/inshokuten`・`/office`は相互リンクのみで入口なし。`/souzoku/taiwan`は親の`/souzoku`からもリンクなし。`/group-home`はトップページQ&A経由でかろうじて到達可。
- **非対称性**：`llms.txt`（AIクローラー向け）には新設6ページ全てが「主要な情報源」と明記済み。一方、人間向け一次導線（ヘッダー/フッター/services）とLINKA辞書（`src/data/linka/site-services.json`）には未登録。**AIには見えるが人には見えない状態**。

## 訂正：本番デプロイに問題はなかった（当初の誤った所見を修正・2026-07-23）

当初、`web_fetch`で`https://luck428.com/services`を確認した際に旧4カード版（賃貸サポート／売買サポート／管理／多言語サポート）が返り、「07-10承認版が本番未反映」と誤って記録していた。**浦松がVercelダッシュボードのスクリーンショットを共有し、これが誤りだったことが判明**：Production Deploymentは`main`最新コミット（`3516dfd`／PR#110）、Status=Ready、サムネイルにも07-10承認済みの3カード版（「承継するには？／物件を探すには？／日本で暮らすには？」）が表示されており、**本番は正しく最新mainを反映済み**。原因はこちら側の`web_fetch`がキャッシュ経由の古い応答を返したためとみられる。Vercel側の対応・確認は不要。以下の実装（`feat/services-nav-pillar-exposure`）も、通常通りpush→PRマージすれば自動的に本番へ反映される。

## 決定事項（浦松確認済み）

1. **ヘッダー**：「サービス」をメガメニュー化（ホバー/タップでカテゴリ別ピラーリンクを展開）。他の4項目（会社概要／コラム／アクセス／お問い合わせ）は変更しない。
2. **グループホーム構成**：`/group-home`（開設の完全ガイド）を導線上の唯一の入口に統一。`/toushi/group-home`（物件探し）は`/group-home`内の「物件を探す」サブ導線として残し、直接のナビ露出はしない。`cross-links.ts`（行政書士側との分離受任リンク）は`/toushi/group-home`向けのまま維持。
3. **進め方**：本ドキュメントは設計書・ワイヤーフレームまで。コード実装は本ドキュメント承認後に別タスクとして着手（TDD・型チェック・デプロイ確認のフローに乗せる）。
4. **優先順位**：/servicesページ・メガメニューとも、シナリオ別ピラー（相続・グループホーム開設・開業支援・外国人）を主役に配置。基本業務（賃貸・売買・管理）は2026-07-10承認済みのコンサルティング型方針を踏襲し、フッターでの導線に留め、/servicesページ本体では前面に出さない。

## カテゴリ設計（ヘッダー・フッター・/services共通の4分類）

ヘッダー／フッター／servicesページで**同一の4カテゴリ**を使い、どこから入っても同じ地図になるようにする。

| カテゴリ | ハブページ | 含まれるピラー |
|---|---|---|
| **① 相続不動産** | `/souzoku` | 総合ガイド`/souzoku`／流れ`/souzoku/nagare`／空き家`/souzoku/akiya`／台湾の相続人`/souzoku/taiwan` |
| **② グループホーム開設** | `/group-home` | 開設の完全ガイド`/group-home`（→内部リンクで物件探し`/toushi/group-home`） |
| **③ 投資・事業用不動産（開業支援）** | `/toushi` | 総合ガイド`/toushi`／介護事業所`/kaigo`／民泊`/minpaku`／飲食店`/inshokuten`／会社設立×オフィス`/office`／社宅・法人契約`/toushi/shataku` |

## 設計①：ヘッダー（メガメニュー）

- トリガー：既存の「サービス」リンクを`サービス ⌄`のボタン化。デスクトップはホバーで展開＋クリックで固定（キーボード操作・スクリーンリーダー対応のため`aria-expanded`とクリックトグルを併用、ホバーのみに依存しない）。
- レイアウト：4カラム（①相続／②グループホーム開設／③投資・事業用／④外国人）。各カラムは見出し（カテゴリ名・ハブページへのリンク）＋子ページのリンクリスト（最大4〜6項目）。
- 下部に1行、区切り線の下に「よくある質問」「相談事例」「料金」を小さめのユーティリティリンクとして配置。
- モバイル：既存のモバイルメニュー（TenantHeader 314〜355行目）内で「サービス」をアコーディオン化。タップで4カテゴリ見出しが展開し、その下に子ページを表示（デスクトップと同じ情報構造をリスト形式で踏襲）。
- 実装コンポーネント案：`TenantLayout.tsx`内に`ServicesMegaMenu`を新設し、`NAV_HREFS`の「サービス」項目をこのコンポーネントに差し替える。カテゴリ定義は`config/`配下に`services-nav.ts`のような単一ソースを作り、メガメニュー・フッター・/servicesページの3箇所から同じ定義を参照する（表示のズレを防ぐ＝保守性優先）。
